# Incident — 事件响应工作流

当此技能被调用时，帮助用户处理生产事件或进行事后复盘。

## 命令

- `/incident new` — 创建新事件
- `/incident review [incident-id]` — 进行事后复盘

## 新事件响应 (`/incident new`)

### 步骤 1: 事件分级

| 级别 | 描述 | 响应时间 | 通知 |
|------|------|---------|------|
| P0 | 全站宕机 | 立即 | 全部 |
| P1 | 核心功能不可用 | 15 分钟 | team-lead |
| P2 | 非核心功能受损 | 1 小时 | 相关团队 |
| P3 | 小问题 | 下一个 sprint | 观察 |

### 步骤 2: 创建事件文档

```markdown
# 事件报告 — [事件 ID]

## 事件信息
- **ID**: INC-2024-001
- **级别**: P1
- **状态**: 🔄 进行中
- **开始时间**: 2024-01-01 14:30 UTC
- **影响**: 支付功能不可用

## 影响范围
- [影响 1]
- [影响 2]

## 时间线
| 时间 | 事件 |
|------|------|
| 14:30 | 监控报警：支付错误率上升 |
| 14:32 | On-call 工程师收到通知 |
| 14:35 | 开始调查 |
| 14:45 | 发现根本原因 |
| 15:00 | 部署修复 |
| 15:05 | 验证修复有效 |
| 15:10 | 事件关闭 |

## 根本原因
[根本原因分析]

## 修复措施
- [ ] 修复 1
- [ ] 修复 2

## 后续行动
| 行动 | 负责人 | 截止日期 |
|------|--------|---------|
| 添加监控告警 | @sre | 2024-01-03 |
| 代码审查加强 | @eng-lead | 2024-01-05 |
```

## 事件复盘 (`/incident review`)

### 步骤 1: 收集信息

1. 读取事件报告
2. 收集相关日志和指标
3. 访谈相关人员

### 步骤 2: 5 Why 分析

```
问题: 支付失败

Why 1: 为什么支付失败？
→ 因为支付服务连接超时

Why 2: 为什么连接超时？
→ 因为数据库连接池耗尽

Why 3: 为什么连接池耗尽？
→ 因为有一个慢查询占用了所有连接

Why 4: 为什么有慢查询？
→ 因为缺少索引

Why 5: 为什么缺少索引？
→ 因为上次 schema 变更后没有评估索引需求

根本原因: 缺乏数据库变更的索引审查流程
```

### 步骤 3: 改进措施

```markdown
## 改进措施

### 立即行动 (72 小时内)
- [ ] 添加缺失索引
- [ ] 增加连接池大小

### 短期改进 (1 周内)
- [ ] 添加慢查询监控
- [ ] 建立数据库变更检查清单

### 长期改进 (1 个月内)
- [ ] 自动化索引建议
- [ ] 数据库变更审批流程
```

## 输出

1. 事件报告
2. 时间线
3. 根本原因
4. 改进措施和负责人
