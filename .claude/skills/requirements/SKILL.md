# Requirements — 需求分析工作流

当此技能被调用时，帮助用户完成需求分析并生成 PRD。

## 工作流程

### 步骤 1: 需求收集

询问或收集：
1. **用户故事**: "作为 [角色]，我希望 [功能]，以便 [收益]"
2. **功能描述**: 详细描述期望行为
3. **非功能需求**: 性能、安全、可用性要求
4. **依赖关系**: 依赖的其他系统或功能
5. **约束条件**: 技术限制、业务规则

### 步骤 2: 创建 PRD 文档

使用以下模板创建 `docs/requirements/[feature-name]-prd.md`：

```markdown
# [功能名称] PRD

## 1. 概述

[一段式总结：功能背景、目标和范围]

## 2. 用户故事

### US-001: [用户故事标题]

**作为** [角色]
**我想要** [功能]
**以便** [收益]

**验收标准**:
- [ ] [标准 1]
- [ ] [标准 2]

### US-002: ...

## 3. 功能需求

### 3.1 [子功能 1]

[详细描述，包括用户交互流程]

### 3.2 [子功能 2]

## 4. 非功能需求

| 类型 | 要求 |
|------|------|
| 性能 | 响应时间 < X ms |
| 可用性 | 可用性 > 99.9% |
| 安全 | [安全要求] |

## 5. 验收标准

### 5.1 功能验收

- [ ] [功能 1 可测试的验收条件]
- [ ] [功能 2 可测试的验收条件]

### 5.2 成功标准

[如何判断功能成功上线]

## 6. 依赖关系

| 依赖 | 状态 | 说明 |
|------|------|------|
| [依赖功能] | 待开发 | [说明] |

## 7. 风险和假设

### 风险
- [风险 1]
- [风险 2]

### 假设
- [假设 1]
- [假设 2]
```

### 步骤 3: 优先级排序

使用 MoSCoW 方法：

- **Must Have (P0)**: [必须有的功能]
- **Should Have (P1)**: [应该有的功能]
- **Could Have (P2)**: [可以有的功能]
- **Won't Have (P3)**: [本次不考虑]

### 步骤 4: 拆分用户故事

将每个用户故事拆分为任务：

```markdown
## US-001 任务分解

- [ ] Task 1: 数据库 Schema 变更
- [ ] Task 2: API 端点实现
- [ ] Task 3: 前端表单组件
- [ ] Task 4: 集成测试
```

## 输出

1. PRD 文档路径
2. 用户故事列表
3. 优先级建议
4. 任务分解 (可选)
