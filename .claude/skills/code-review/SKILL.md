# Code Review — 代码审查工作流

当此技能被调用时，帮助用户进行代码审查。

## 工作流程

### 步骤 1: 收集信息

1. 获取 PR 信息或指定文件
2. 读取代码变更
3. 了解相关需求和设计文档

### 步骤 2: 执行代码审查

#### 审查清单

**正确性**
- [ ] 代码是否正确实现需求？
- [ ] 边界条件是否处理？
- [ ] 错误处理是否完善？

**可读性**
- [ ] 代码是否清晰易懂？
- [ ] 命名是否有意义？
- [ ] 是否有适当的注释？

**可测试性**
- [ ] 是否有单元测试？
- [ ] 测试覆盖率是否足够？
- [ ] 测试是否可重复执行？

**安全性**
- [ ] 是否有 SQL 注入风险？
- [ ] 是否有 XSS 风险？
- [ ] 敏感数据是否加密？
- [ ] 是否有硬编码的秘密？

**性能**
- [ ] 是否有性能问题？
- [ ] 数据库查询是否优化？
- [ ] 是否有 N+1 查询？

**最佳实践**
- [ ] 遵循语言/框架最佳实践？
- [ ] 代码是否 DRY？
- [ ] 依赖是否合理？

### 步骤 3: 生成审查报告

```markdown
# 代码审查报告

## PR 信息
- **PR**: [#123] 用户登录功能
- **作者**: @developer
- **审查者**: [审查者]
- **日期**: YYYY-MM-DD

## 变更摘要
[变更内容概述]

## 审查结果

### ✅ 通过
- [项 1]
- [项 2]

### ⚠️ 需要修改
- [问题 1] — [建议]
- [问题 2] — [建议]

### ❌ 阻塞问题
- [严重问题] — [必须修改]

## 详细评论

### 文件: src/backend/auth.py

**行 45**: 硬编码的 API Key
```python
# 错误
api_key = "sk-1234567890"

# 建议
api_key = os.environ.get("API_KEY")
```

## 总结

| 类别 | 结果 |
|------|------|
| 正确性 | ✅ 通过 |
| 可读性 | ⚠️ 需要修改 |
| 安全性 | ❌ 阻塞 |
| 性能 | ✅ 通过 |
| 测试 | ⚠️ 需要修改 |

**总体结论**: ❌ 需要修改后重新审查
```

## 输出

1. 审查报告
2. 详细评论
3. 修改建议
4. 总体结论 (通过/需要修改/阻塞)
